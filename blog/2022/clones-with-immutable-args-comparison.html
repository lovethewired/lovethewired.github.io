Clones With Immutable Args

Gas Comparisons

23 Oct 2022

Clones with immutable args (CWIA) aim to save gas by avoiding storage reads and writes.

Clones - and proxies alike - usually contain a initializer function to set parameters specific to the clone. This is due to the nature of contracts that function as proxies. Typically they use standardized and optimized implementations that contain minimal logic and forward all calls to an contract that handles the implementation logic.

Immutable variables are useful, because instead of writing these variables into the contract's storage space, they are directly included in the contract's runtime bytecode. They work in a similar way as variables that are declared to be constant, but they are evaluated and inserted during contract creation - in the constructor.

Since solidity does not allow for dynamic type immutable variables, it's not that straightforward to write a clone/proxy contract that will just take in a immutable string of bytes to be easily accessed in the implementation contract.

Another reason for the slow adoption of clones that use immutable args is that they are often heavily optimized. EIP-1167 formalizes the specs for minimal proxy clones.

Because EIP-1667 clones can easily be identified by checking their bytecode, services such as etherscan are able to detect these. If the contract were to include immutable args however, this would require more rules in order to identify such contracts, because the immutable args must be included in the contract's runtime code.

Clones with immutable args - via CALL

To illustrate some of the ideas behind clones with immutable args, we can think of a minimal clone contract that takes in an immutable arg as a parameter.

contract CloneWithImmutableArgsVer1 {
    address private immutable logic;
    uint256 public immutable immutableArg0;

    constructor(address logic_, uint256 immutableArg0_) payable {
        logic = logic_;
        immutableArg0 = immutableArg0_;
    }

    fallback() external payable {
        address logic_ = logic;

        assembly {
            calldatacopy(0, 0, calldatasize())

            let success := delegatecall(gas(), logic_, 0, calldatasize(), 0, 0)

            returndatacopy(0, 0, returndatasize())

            if success {
                return(0, returndatasize())
            }

            revert(0, returndatasize())
        }
    }
}

This contract takes in a address logic_ to which all calls are delegated to and a uint256 immutableArg0_ that is accessible to the clone as a public function.

There is still one problem, however. We usually will want to access immutableArg0 from within the logic contract. Since the logic contract does not know about immutableArg0 - let alone the clone contract itself - it is not directly available to the implementation. Although, since we know that we will be calling the logic contract via delegatecall, we can call back into the clone contract via CloneWithImmutableArgsVer1(address(this)).immutableArg0() to access the variable.

Essentially this method will result in an additional call operation and a codecopy operation once the context is set back to the clone contract. staticcall could also be used instead to ensure that no persistent changes are being made.

Another problem is that the current contract only works for one fixed 32 bytes word. Ideally, we would be able to store and access arbitrary byte strings. A way to get around this is shown in the next possible method that uses extcodecopy.

Clones with immutable args - via EXTCODECOPY

Instead of calling back into the clone contract, we could also directly access the bytes via extcodecopy while remaining in the context of the implementation contract. This example will work with immutable args that can be bytes of arbitrary lengths.

contract CloneBase {
    fallback() external payable {
        address logic = ImmutableArgsVer2Lib.getLogicAddress();

        assembly {
            calldatacopy(0, 0, calldatasize())

            let success := delegatecall(gas(), logic, 0, calldatasize(), 0, 0)

            returndatacopy(0, 0, returndatasize())

            if success {
                return(0, returndatasize())
            }

            revert(0, returndatasize())
        }
    }
}

contract ClonesWithImmutableArgsVer2 {
    constructor(address logic, bytes memory immutableArgs) payable {
        bytes memory code = abi.encodePacked(
            type(CloneBase).runtimeCode,
            immutableArgs,
            uint16(immutableArgs.length),
            logic
        );

        assembly {
            return(add(code, 0x20), mload(code))
        }
    }
}

The delegatecall logic in the fallback function remains the same, except that we are now reading the address of the logic contract using a library function ImmutableArgsVer2Lib.getLogicAddress() - which we'll get into in a bit.

Out goal is to append additional data - the immutable args - to the contract bytecode so that we can cheaply access these later. ClonesWithImmutableArgsVer2 acts as a kind of "wrapper" contract that returns the code of the underlying contract - CloneBase - with additional data appended - the immutable args, the length of the immutable args and the logic contract.

Since we're accessing type(CloneBase).runtimeCode we need to avoid any immutable keywords, because these would not be available in the runtime code. That's why we also append the logic address to the contract code to be able to access this in the fallback function of the clone.

We now need to write the functions that will make it possible to retrieve address logic and bytes immutableArgs from the contract code.

 library ImmutableArgsVer2Lib {
    function getLogicAddress() internal returns (address logic) {
        assembly {
            // `address logic` is encoded into bytecode at last position.
            // Load 20 bytes (0x14) starting from end - 20 bytes.
            let offsetLogic := sub(codesize(), 0x14)
            // Store in scratch space in first slot at 12 bytes (0x0c = 32 - 20) offset.
            codecopy(0x0c, offsetLogic, 0x14)
            // Load first memory slot to retrieve the address.
            logic := mload(0)
        }
    }

    function getImmutableArgs() internal returns (bytes memory args) {
        assembly {
            // Clear dirty scratch space.
            mstore(0, 0)

            let codeSize := extcodesize(address())
            // The offset for the length of the args is stored
            // at `codeSize - 20 - 2` (20 bytes address + 2 bytes length)
            let argsLenOffset := sub(codeSize, 0x16)

            // Load length of immutable args from contract bytecode.
            extcodecopy(address(), 0x1e, argsLenOffset, 0x02)

            let argsLen := mload(0)
            let argsOffset := sub(argsLenOffset, argsLen)

            // Place `args` at free memory position.
            args := mload(0x80)
            // Store length of `args` bytes-array.
            mstore(args, argsLen)

            // Copy immutable args to `args` memory offset.
            extcodecopy(address(), add(args, 0x20), argsOffset, argsLen)

            // Advance the `memPtr` by `argsLen + 0x20`,
            // rounded up to the next multiple of 32.
            mstore(0x80, shl(5, shr(5, add(add(args, argsLen), 0x3f))))
        }
    }
}

Notice how getLogicAddress() uses codecopy and how getImmutableArgs() uses extcodecopy. This is because getLogicAddress() will be executed in the context of the clone - meaning the address is stored in the current runtime code - whereas getImmutableArgs() will be executed in the context of the logic contract, which is unable to access the immutable args stored in the runtime code and thus must use extcodecopy.

Clones with immutable args - via CALLDATA

The original clones with immutable args uses another trick. The immutable args are directly appended to the calldata of the delegatecall in the clone's fallback function.

CloneWithImmutableArgsVer3's constructor code remains the same as CloneWithImmutableArgsVer2.

contract CloneWithImmutableArgsVer3 {
    constructor(address logic, bytes memory immutableArgs) payable {
        bytes memory code = abi.encodePacked(
            type(CloneBaseVer3).runtimeCode,
            immutableArgs,
            uint16(immutableArgs.length),
            logic
        );

        assembly {
            return(add(code, 0x20), mload(code))
        }
    }
}

Now when we make the delegatecall, we need to concatenate the the following data in memory: calldata + immutable args + uint16 length of immutable args. Retrieving and copying the immutable args works exactly the same as with the previous version, except that, since we're in the clone's context, we can use codesize and codecopy instead of extcodesize and extcodecopy.

contract CloneBaseVer3 {
    fallback() external payable {
        address logic = ImmutableArgsVer2Lib.getLogicAddress();

        assembly {
            // Clear dirty scratch space.
            mstore(0, 0)

            // Get length and offset of immutable args.
            let argsLenOffset := sub(codesize(), 0x16)
            codecopy(0x1e, argsLenOffset, 0x02)

            let argsLen := mload(0)
            let argsOffset := sub(argsLenOffset, argsLen)

            calldatacopy(0, 0, calldatasize())

            // Copy immutable args to end of calldata in memory.
            codecopy(calldatasize(), argsOffset, argsLen)
            // Append the length of the immutable args to the end.
            mstore8(add(calldatasize(), argsLen), shr(8, argsLen))
            mstore8(add(add(calldatasize(), argsLen), 1), argsLen)

            let success := delegatecall(gas(), logic, 0, add(add(calldatasize(), argsLen), 2), 0, 0)

            returndatacopy(0, 0, returndatasize())

            if success {
                return(0, returndatasize())
            }

            revert(0, returndatasize())        
        }
    }
}

Now the implementation contract will have direct access to the immutable args via calldata. This part is left out, because it works in pretty much the same way as retrieving immutable args from the contract code.

This method has the benefit that it doesn't need to make another external call or copy external code (since it uses codecopy instead). Accessing the immutable args via calldata is cheap and almost practically negligible - except for when this is run on rollups that commit calldata to the root chain (here calldata makes up for over 50% of the transaction cost). Yet, unlike before, where we only access the immutable args when needed, these are always appended to any function call - no matter if they are required or not.

Tradeoffs and Gas Comparisons

The full code for all comparisons can be found here https://github.com/0xPhaze/clones-with-immutable-args-comparison (though method 1 and method 3 are swapped here). The clone implementations are written in pure bytecode to better understand the limits. The methods that use codecopy could however, still be further optimized to not use codecopy at all, but to loop over push opcodes that push the data directly onto the stack and store it in memory.

Relevant OpcodesNotes
Method 1call + codecopyCalls back into clone contract to retrieve immutable arg.
Method 2extcodecopyLoads external code from clone contract.
Method 3codecopy + calldataloadAdds a gas overhead to EVERY function call
